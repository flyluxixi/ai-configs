# PHP/Webman 项目开发规范

最后更新：2026-04-21

## 最高原则

性能和稳定性高于一切。具体体现为：

- **能让平台解决的，不用代码绕**：PHP 新特性、PostgreSQL 高级查询、Redis 原子命令、Nginx 内置能力，优先于自己堆逻辑
- **能一次解决的，不做两次**：N+1 查询、重复计算、多次 IO 能合并的必须合并
- **跑通不等于正确**：没有内存泄漏、没有垃圾数据、没有资源残留，才算完成

遇到多种实现方案时，以上面三条为判断标准，选资源开销最小、执行路径最短的那个。

## 技术栈与环境

- **OS**: AlmaLinux 10.1 / Ubuntu 24.04，纯 Linux 命令行部署，无 Docker
- **主力栈**: PHP（当前生产版本为准）+ Webman（常驻内存进程，非 FPM）
- **数据库**: PostgreSQL
- **缓存 / 队列**: Redis
- **Web Server**: Nginx

> PHP 版本以实际部署环境为准，不在此处写死。生成代码前先确认当前版本，使用该版本已正式发布的特性，不使用 RC / alpha 阶段的功能。

根据项目需求选最优方案，不强制绑定技术栈；若有明显更优选择，直接说明理由推荐。引入新系统级依赖需说明理由和运维影响。

## 优先使用平台能力

**遇到以下情况，必须在回复中说明为什么无法使用平台能力，不得静默绕开：**
- 打算用 PHP 循环处理本可以一条 SQL 批量完成的操作
- 打算用多次查询组装数据，而不是 CTE / 窗口函数一次返回
- 打算手动实现计数、限流、锁，而不是用 Redis 原子命令
- 打算引入未经确认的新 Composer 包

**必须优先使用的平台特性（不是可选项）：**

| 平台 | 应该用 | 不该绕开 |
|---|---|---|
| PHP | 优先使用当前版本支持的语言特性（fiber、match、enum、readonly、named arguments、property hooks 等）| 旧式 array_map 链、手动类型转换 |
| PostgreSQL | CTE、窗口函数、JSONB、upsert（ON CONFLICT）、全文搜索 | PHP 层拼装分页、循环 insert |
| Redis | INCR/DECR、pipeline、lua script、SETNX、stream | PHP 层手动加锁、循环 get/set |
| Nginx | upstream 缓存、limit_req、gzip、try_files | PHP 层做静态文件处理 |

## PHP 编码规范

### 基础要求

- 所有文件顶部必须声明 `declare(strict_types=1);`
- 积极使用当前 PHP 版本的新特性，但不为新特性而新特性
- 命名空间必须与目录结构严格对应
- 禁止魔法数字，用常量或枚举替代
- 日志按天切割（logrotate），保留 30 天

### 命名规范

| 场景 | 风格 | 示例 |
|---|---|---|
| 类名 | PascalCase | `OrderService` |
| 方法 / 变量 | camelCase | `createOrder`, `$userId` |
| 常量 | UPPER_SNAKE_CASE | `MAX_RETRY_COUNT` |
| 数据库字段 | snake_case | `created_at` |
| 文件 / 目录 | 与类名一致 | `OrderService.php` |

### 类型声明

优先级：`具体类型` > `?nullable` > `A|B union` > `mixed`

`mixed` 仅用于真正无法约束类型的场景（如序列化、泛型容器）。

### 代码结构

- 单个方法行数控制在 50 行内；超过时说明原因，给出拆分方案，确认后再执行
- 嵌套不超过 3 层；超过则直接提取方法，无需确认
- 单个类不超过 300 行；超过时说明原因，给出拆分方案，确认后再执行
- **严禁在循环内执行数据库查询**

### 错误处理

- 业务错误抛出自定义异常，不用 `false` / `null` 表示失败
- 所有外部调用（HTTP、数据库、Redis）必须有异常捕获
- 日志要包含足够上下文（请求 ID、用户 ID、关键参数）
- 生产环境禁止将详细错误信息暴露给客户端

### Webman 常驻内存——高风险区域

Webman 不同于 FPM，进程常驻，以下错误会导致内存持续增长或数据污染，**必须严格避免**：

**内存泄漏（禁止）：**
- 禁止用静态属性缓存请求级数据：`static $cache = []` 跨请求累积，永远不会释放
- 禁止在 Worker 启动后重复注册事件监听器（每个请求注册一次 = 内存线性增长）
- 禁止在请求处理中创建未归还的数据库连接，连接必须由框架连接池管理
- 禁止手动 new PDO

**数据污染（禁止）：**
- 禁止用 `$GLOBALS` 传递请求数据，请求级数据只通过 `$request` 传递
- 禁止在静态变量或单例中存储任何与当前请求相关的状态

**资源残留（必须处理）：**
- 请求结束后必须手动清理非框架托管的资源（临时文件、非池化连接、外部句柄）
- 事务失败必须回滚，不能留下半写入的关联数据

## 数据库规范

- SQL 优先写原生查询或 Query Builder，不强制 ORM
- PostgreSQL 特性可放心使用：JSONB、数组类型、CTE、窗口函数、全文搜索、upsert
- **禁止字符串拼接 SQL**，所有查询必须参数绑定
- 迁移脚本写成幂等形式，可重复执行
- **禁止在循环内执行查询**，改用批量查询或 IN 语句
- 软删除逻辑必须一致，定期清理孤儿记录，不允许垃圾数据长期累积

---

## 队列 / 异步任务规范

- 所有队列任务必须实现幂等性，通过唯一业务 key 防止重复执行
- 必须定义重试策略：最大重试次数、退避间隔（建议指数退避）
- 必须配置死信处理：超出重试次数后转入死信队列并告警，不能静默丢弃
- 任务必须设置执行超时，防止僵尸任务长期占用 worker
- 任务内禁止嵌套分发新任务

---

## 安全规范

- 密钥、appid、appsecret 等全部走**环境变量**，禁止硬编码
- 所有用户输入必须验证和过滤；所有输出到 HTML 的内容必须转义，防止 XSS
- 文件上传必须校验 MIME 类型和扩展名，禁止存储到 Web 可访问路径
- 涉及状态变更的接口必须有 CSRF 防护
- 对外接口默认：参数校验 + 频率限制（Redis 计数）
- 敏感数据（密码、密钥、完整 token）不写入日志
- 涉及金额的操作必须有幂等性保护
- 除完全公开接口外，API 必须有认证和授权检查
- 其他 Web 安全问题遵循 OWASP Top 10，具体防护措施按项目要求补充

---

## 配置管理

- **配置与代码分离**：运行时配置（端口、进程数、超时、日志级别等）一律通过环境变量注入，`config/` 文件只写结构和默认值，不写环境差异
- `config/` 文件进 git，`.env` 不进 git；`.env.example` 提供所有变量的说明和安全默认值
- 禁止在 `config/` 文件中直接硬编码因服务器而异的值（如具体端口号、IP 地址、进程数）
- 敏感配置（密钥、密码、token）必须走环境变量，禁止硬编码（见安全规范）

## 行为规范

- 复杂任务（涉及多文件修改、架构变更）动手前先说明方案，确认后再执行
- 单文件小修改直接执行，完成后说明改了什么
- 上述之外显而易见的操作不要反复询问
- 需求不明确时直接问，不要自行假设
- 若任务要求本身存在明显的设计缺陷或性能风险，直接说明并提出替代方案，不沉默执行
- 接到多步骤任务时，先列出每步的完成标准（不是执行步骤），确认后再动手
- 每次只修改任务范围内的文件，不顺手重构无关模块
- 完成后说明改了什么、为什么、有何注意事项、用了哪些平台特性；不罗列执行步骤，直接给结论
- 发现潜在问题（性能隐患、安全漏洞、逻辑缺陷）主动提出
- 遇到多种可行方案，直接推荐最优的一种并说明理由，不做冗长对比
- 代码不省略关键实现、不用 `// ...` 占位、不用"其余保持不变"之类模糊表述
- 给出 Nginx / Redis / PostgreSQL 配置时，提供完整配置块，不给片段
- **不自动生成测试文件**，除非明确要求
- 代码之外的解释保持简洁，不重复已知信息
- **本项目文件修改后必须询问是否同步全局**：本项目是 `~/.claude/` 的源仓库，`skills/`、`agents/`、`scripts/`、`commands/`、`rules/`、`settings.json` 等目录和文件在 `~/.claude/` 中均有对应路径；每次修改项目版本后，必须询问用户是否覆盖 `~/.claude/` 中的对应文件，不得自动同步
