# Go / Gin 规则

适用范围：Go + Gin 项目。数据库、缓存、Web Server 的专项规则分别见 `postgresql.md`、`redis.md`、`nginx.md`。

## 技术栈与环境

- Go（当前项目 `go.mod` 声明版本为准）+ Gin

生成代码前先读取 `go.mod`、`go.sum` 和项目启动入口，确认 Go 版本、模块路径、已使用的框架/库和项目目录结构。

## Go 平台能力

遇到以下情况，必须在回复中说明为什么无法使用 Go 标准库或项目已有能力，不得静默绕开：

- 打算新增第三方包处理标准库已能稳定解决的问题
- 打算手写全局状态、缓存、任务调度或连接管理，而不是复用项目已有封装
- 打算绕过 `context.Context` 实现超时、取消、请求级追踪
- 打算在应用层循环处理本可以由数据库批量完成的操作；涉及数据库批量处理时同时遵守 `postgresql.md`

## 禁止事项

- 禁止忽略 `error`；不能用 `_` 丢弃错误，除非同一行注释说明原因
- 禁止在 handler / service 中 `panic` 表示业务失败；业务错误必须返回明确错误并由统一错误处理中间件转换响应
- 禁止把 `gin.Context` 存入 goroutine、全局变量、结构体字段或请求结束后继续使用；异步任务只传必要值和 `context.Context`
- 禁止在请求处理中创建长期生命周期资源（数据库连接池、Redis 客户端、HTTP client 等）；这类资源必须在启动阶段初始化并复用
- 禁止每次请求创建未设置超时的外部 HTTP client；外部调用必须有超时、取消和错误处理
- 禁止无边界 goroutine：启动 goroutine 必须有退出条件、错误处理和必要的 recover
- 禁止在 goroutine 中写入 `http.ResponseWriter` / `gin.Context` 响应对象
- 禁止直接拼接 SQL；所有查询必须参数绑定，涉及数据库规则同时遵守 `postgresql.md`
- 禁止硬编码密钥、appid、appsecret、token；统一走环境变量或项目配置规范
- 禁止未经确认新增 Go module、系统包或外部服务依赖
- 禁止在日志中输出密码、密钥、完整 token、验证码等敏感信息
- 禁止用全局可变变量保存请求级状态、当前用户、trace id、租户 id 等上下文

## 编码规范

- 包名短小、全小写，不使用下划线或驼峰
- 导出标识符必须有必要注释；非导出代码避免无意义注释
- 错误返回值放在最后一个返回值
- 业务错误应有可判断的类型或哨兵错误，避免只靠字符串匹配
- 不使用魔法数字；用常量或配置项表达业务含义
- 小函数优先；函数过长或分支过深时提取私有函数
- 修改公共结构体、接口、路由、中间件时同步检查调用方和测试覆盖

## Gin 规范

- handler 只负责参数绑定、鉴权上下文读取、调用 service、返回响应；复杂业务逻辑放到 service 层
- 请求参数必须显式校验，不依赖零值表示合法输入
- 响应格式应遵循项目既有结构，不自行新增另一套 response wrapper
- 中间件只处理横切逻辑（认证、日志、trace、recover、限流等），不写业务流程
- 路由分组、版本前缀、鉴权边界按项目既有风格组织，不自行另起风格
- `gin.Context` 中只放请求级轻量值，不放大对象、连接、client 或可变业务状态

## Context 与并发

- 函数涉及 I/O、数据库、Redis、HTTP 调用时优先接收并传递 `context.Context`
- 外部调用必须设置超时，不能无限等待
- goroutine 必须能随请求取消、服务关闭或任务完成而退出
- 共享数据必须用 channel、mutex 或其他明确同步机制保护，禁止数据竞争
- 后台任务需要定义幂等 key、重试策略、超时和失败处理

## 依赖管理

- 新增第三方 Go module 前必须说明理由，等确认后再写入 `go.mod`
- 优先使用项目已有库和标准库
- 新增系统级依赖、外部服务或运行时组件时，需说明运维影响

## 文档查询

- 查询 Gin 或第三方 Go module API 时，优先使用官方文档、项目 `go.mod` 锁定版本或源码
- 不凭记忆假设 Gin、中间件、ORM、配置库的具体 API 或行为
