# PostgreSQL 规则

适用范围：PostgreSQL schema、迁移、查询、索引、事务、性能排查，以及业务代码中生成或调用 PostgreSQL 查询的场景。

## PostgreSQL 能力

遇到以下情况，必须在回复中说明为什么无法使用 PostgreSQL 能力，不得静默绕开：

- 打算在应用层分页、聚合、排序或去重，而不是让 SQL 一次返回结果
- 打算用多次查询组装数据，而不是 CTE / 窗口函数一次返回
- 打算循环 insert / update，而不是批量写入或 upsert
- 打算先查再插实现唯一写入，而不是用唯一约束 + `ON CONFLICT`

必须优先使用的平台特性：

| 应该用 | 不该绕开 |
|---|---|
| CTE | 应用层多轮拼装中间结果 |
| 窗口函数 | 应用层手动排名、分组计数 |
| JSONB | 字符串存 JSON 后应用层解析 |
| upsert（`ON CONFLICT`） | 先查再插导致竞态 |
| 全文搜索 | 简单粗暴的应用层模糊过滤 |
| 事务 | 多表写入靠应用层补偿 |

## 禁止事项

- 禁止字符串拼接 SQL；所有查询必须参数绑定
- 禁止在循环内执行查询或写入；改用批量查询、`IN`、JOIN、CTE、窗口函数、批量写入或 upsert
- 禁止先查再插实现唯一写入；必须使用唯一约束 + `ON CONFLICT`
- 禁止无事务地执行多表一致性写入、金额变更、库存扣减、状态流转等操作
- 禁止在事务内执行 HTTP、Redis、文件 IO 等不可控外部调用
- 禁止迁移脚本不可重复执行；迁移必须幂等或由迁移框架保证只执行一次
- 禁止无索引地对大表做模糊查询、排序、过滤或 JOIN
- 禁止为了“可能会用到”盲目加索引；新增索引必须对应明确查询
- 禁止忽略慢查询、锁等待、执行计划异常或表膨胀问题

## 查询规范

- SQL 优先保证可读、可解释、可优化；复杂查询可使用原生 SQL 或 Query Builder，不为 ORM 形式牺牲 SQL 质量
- 查询字段应明确列出，避免无必要的 `SELECT *`
- 需要分页时优先让数据库完成排序和分页；大偏移分页应评估 keyset pagination
- 批量读取使用集合查询，不用应用层逐条查
- 软删除逻辑必须一致，定期清理孤儿记录，不允许垃圾数据长期累积

## 事务规范

- 多表写入、金额、库存、状态流转、幂等记录写入必须有明确事务边界
- 事务内只放数据库一致性相关操作，避免长事务
- 事务失败必须回滚，不能留下半写状态
- 并发更新同一业务对象时，必须说明使用的并发控制方式（唯一约束、行锁、乐观锁、upsert 等）
- 使用行锁时说明锁定范围，避免扩大锁粒度

## 索引规范

- 查询条件、JOIN key、排序字段、唯一性约束应评估索引
- 新增索引必须说明服务的查询或约束
- 大表新增索引需考虑锁表和线上写入影响，必要时使用 `CREATE INDEX CONCURRENTLY`
- 不盲目创建重复索引、低选择性索引或长期无人使用的索引
- JSONB 查询需要评估合适的 GIN / 表达式索引

## 性能诊断

- SQL 性能问题先看 `EXPLAIN (ANALYZE, BUFFERS)`，不凭感觉改 SQL 或加索引
- 关注是否出现不合理顺序扫描、排序、临时文件、回表、Nested Loop、Hash Join 或锁等待
- 优化前先确认数据量级、选择性、现有索引和实际查询参数
- ORM / QueryBuilder 生成的 SQL 也按真实 SQL 审查

## 迁移与数据维护

- schema 迁移必须可审查、可回滚或有明确补救方案
- 数据迁移要分批执行，避免长事务和大表长时间锁定
- 新增约束前先确认历史数据满足约束
- 删除字段、表或索引前必须确认调用方和历史任务不再依赖
- 定期关注慢查询、表/索引膨胀、无用索引和孤儿数据
