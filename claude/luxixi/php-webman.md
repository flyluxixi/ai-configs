# PHP / Webman 规则

适用范围：PHP + Webman 项目。数据库、缓存、Web Server 的专项规则分别见 `postgresql.md`、`redis.md`、`nginx.md`。

## 技术栈与环境

- PHP（当前生产版本为准）+ Webman（常驻内存进程，非 FPM）

PHP 版本以实际部署环境为准，不在规则中写死。生成代码前先确认当前版本，使用该版本已正式发布的特性，不使用 RC / alpha 阶段的功能。

根据项目需求选最优方案，不强制绑定技术栈；若有明显更优选择，直接说明理由推荐。引入新系统级依赖需说明理由和运维影响。

## PHP 平台能力

遇到以下情况，必须在回复中说明为什么无法使用平台能力，不得静默绕开：

- 打算用 PHP 循环处理本可以由数据库批量完成的操作；涉及数据库批量处理时同时遵守 `postgresql.md`
- 打算引入未经确认的新 Composer 包
- 打算在请求路径中创建可复用的重量级对象，而不是交给容器或框架生命周期管理

必须优先使用当前 PHP 版本支持的语言特性，但不为新特性而新特性。

## PHP 编码规范

### 基础要求

- 所有 PHP 文件顶部必须声明 `declare(strict_types=1);`
- 命名空间必须与目录结构严格对应
- 禁止魔法数字，用常量或枚举替代
- 日志轮转按项目日志体系执行；Webman 日志初始化参考 `/d-webman-log`

### 命名规范

| 场景 | 风格 | 示例 |
|---|---|---|
| 类名 | PascalCase | `OrderService` |
| 方法 / 变量 | camelCase | `createOrder`, `$userId` |
| 常量 | UPPER_SNAKE_CASE | `MAX_RETRY_COUNT` |
| 数据库字段 | snake_case | `created_at` |
| 文件 / 目录 | 与类名一致 | `OrderService.php` |

### 类型声明

优先级：`具体类型` > `?nullable` > `A|B union` > `mixed`

`mixed` 仅用于真正无法约束类型的场景（如序列化、泛型容器）。

### 代码结构

- 单个方法行数控制在 50 行内；超过时说明原因，给出拆分方案，确认后再执行
- 嵌套不超过 3 层；超过则直接提取方法，无需确认
- 单个类不超过 300 行；超过时说明原因，给出拆分方案，确认后再执行
- 严禁在循环内执行数据库查询

### 错误处理

- 业务错误抛出自定义异常，不用 `false` / `null` 表示失败
- 所有 PHP 侧 HTTP / 外部服务调用必须有异常捕获
- 日志要包含足够上下文（请求 ID、用户 ID、关键参数）
- 生产环境禁止将详细错误信息暴露给客户端

## Webman 常驻内存高风险区域

Webman 不同于 FPM，进程常驻，以下错误会导致内存持续增长或数据污染，必须严格避免。

先判断代码是在 Worker 启动时执行一次，还是每请求执行；生命周期判断直接决定是否存在内存泄漏或跨请求状态污染。

### 内存泄漏

- 禁止用静态属性缓存请求级数据：`static $cache = []` 跨请求累积，永远不会释放
- 禁止在 Worker 启动后重复注册事件监听器（每个请求注册一次 = 内存线性增长）
- 禁止在请求处理中创建未归还的数据库连接，连接必须由框架连接池管理
- 禁止手动 `new PDO`
- 容器单例不能持有请求级引用（`$request`、当前用户上下文等）

### 数据污染

- 禁止用 `$GLOBALS` 传递请求数据，请求级数据只通过 `$request` 传递
- 禁止在静态变量或单例中存储任何与当前请求相关的状态
- `session_start`、`ini_set`、`putenv` 等全局副作用会污染后续请求，禁止在请求处理中随意调用

### 资源残留

- 请求结束后必须手动清理非框架托管的资源（临时文件、非池化连接、外部句柄）
- 事务失败必须回滚，不能留下半写入的关联数据

## 依赖管理

- 引入新的第三方 Composer 包前必须说明理由，等确认后再写入代码
- 系统级新依赖（系统包 / PHP 扩展，注明适用 apt 还是 dnf）需同时说明运维影响
