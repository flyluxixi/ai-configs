---
name: d-step
description: 项目原子化拆解与自动执行 skill。将复杂项目拆解为最小可独立执行的原子任务，生成 STEP.md 后自动连续执行，只在关键节点暂停等待确认。适用于 Claude Code、Codex 等 AI 编程工具。触发条件：用户说"开始执行"、"拆解并执行"、"帮我自动完成"、"/d-step"，或用户明确授权 AI 自动完成整个功能开发时。仅想查看任务拆解结构但暂不执行时不触发。单文件修改、简单 bug 修复、单步任务不触发。
---

把「把整个项目扔给 AI」变成「每次只给 AI 一个刚好够的任务，然后自动跑完」。

---

## Step 1：检查现有文件状态

**① 检查 `STEP.md` 是否存在：**
- **存在且有效** → 读取文件，找到第一个未完成的 Task，告知用户：
  ```
  📋 发现已有 STEP.md，当前进度：
  已完成：Task X.X ~ X.X
  下一个：Task X.X [任务名]

  直接继续执行，还是重新拆解？
  ```
  - 用户选择继续执行 → 跳到 **Step 5：自动执行**
  - 用户选择重新拆解 → 删除 STEP.md，继续 Step 2

**② 检查 `STEP_DRAFT.md` 是否存在（断点续拆）：**
- **存在** → 读取草稿，告知用户：
  ```
  📋 发现未完成的拆解草稿，已拆解模块：[已完成的模块列表]

  从断点继续，还是丢弃草稿重新拆解？
  ```
  - 用户选择继续 → 从断点继续拆解，完成后进入 Step 5
  - 用户选择丢弃重拆 → 删除 STEP_DRAFT.md，继续 Step 2

**③ 两个文件都不存在** → 继续 Step 2。

---

## Step 2：获取需求

优先读取 `ASK.md`。同时读取 `CLAUDE.md`、`README.md`、`PROJECT_BACKGROUND.md` 补充上下文。

**如果存在 `ASK_CHANGELOG.md`**，读取并检查是否有「待评估」条目：

- **有「待评估」条目** → 停止，告知用户：
  ```
  ⚠️ ASK_CHANGELOG.md 中有待评估的需求变更，必须先处理后再拆解：
  [列出待评估条目]

  请通过 d-ask 将变更合并、拒绝或标记为延期，更新 ASK.md 后再运行 d-step。
  ```
- **无「待评估」条目** → 继续拆解。

**如果 `ASK.md` 不存在：**
```
⚠️ 未找到 ASK.md。建议先运行 d-ask 完成需求确认，拆解结果会更准确。
是否继续仅凭当前信息拆解？
```
若用户选择继续，补问最小必要信息：交付物、技术栈、主要功能模块。

---

## Step 3：识别模块边界

按「高内聚低耦合」划分功能域：

1. 列出所有功能点（格式：动词 + 名词，如「创建用户」「发送通知」）
2. 按业务域聚合成模块
3. 标注模块间依赖关系：

```
模块 A ──► 模块 B ──► 模块 D
           模块 C ──► 模块 D
```

---

## Step 4：原子化拆解，写入 STEP.md

每个 Task 必须同时满足：
- **单一职责**：只做一件事
- **单次对话可完成**：AI 在一个上下文窗口内能独立完成
- **输出可独立验证**：有明确完成标志（「通过了 XX 测试」而不是「写完了」）

粒度参考：
- ❌ 太粗：「实现用户系统」
- ❌ 太细：「写一个 if 语句」
- ✅ 合适：「实现 JWT 登录接口，返回 token，错误时返回 401」

拆解过程中每完成一个模块，写入 `STEP_DRAFT.md` 保存进度。

全部模块拆解完成后，写入 `STEP.md`：

```markdown
# 项目拆解文档

> 由 d-step 生成。
> 最后更新：[日期时间]
> 当前执行进度：Task 0/N

## 模块总览
| 模块 | 描述 | 依赖 | 顺序 |
|------|------|------|------|
| auth | 用户认证 | — | 1 |
| user | 用户管理 | auth | 2 |

## 依赖关系
auth ──► user ──► notify

## 模块详情

### [模块名]
**职责：**[一句话]

**任务清单：**
- [ ] Task 1.1：[名称] — 输入：X，输出：Y，验收：Z
- [ ] Task 1.2：[名称] — 输入：X，输出：Y，验收：Z

**接口契约：**[暴露给其他模块的数据结构或函数签名]

---

## 执行顺序
推荐顺序：Task 1.1 → Task 1.2 → Task 2.1 → ...
可并行：[无依赖关系、可同时执行的任务]

## 暂停条件（遇到以下情况才停下来等待确认）
- 删除或覆盖已有重要文件
- 遇到真正的技术分叉（两种方案都合理，需要人决策）
- Task 验收失败且尝试修复 2 次后仍失败
- 上下文窗口接近上限（主动告知用户，保存进度，等待新对话）
```

STEP.md 完整写入后，开始执行第一个 Task 之前，删除 `STEP_DRAFT.md`。

---

## Step 5：自动执行

写完 STEP.md 后（或从断点继续时），**不停下来等待，直接开始执行**。

### 执行循环

```
读取 STEP.md → 找第一个未完成 [ ] Task
→ 执行 Task
→ 验收（运行测试 / 检查输出 / 验证接口）
→ 验收通过：将 [ ] 改为 [x]，更新 STEP.md 中的执行进度
→ 找下一个未完成 Task → 继续循环
→ 所有 Task 完成 → 输出完成报告
```

### 执行时的行为规范

**核心约束：禁止重新定义 ASK.md 中已确认的架构与边界。**
ASK.md 是需求基准，d-step 的职责是实现它，而不是优化或改进它。
如果执行过程中发现 ASK.md 的某个决策存在问题，必须暂停并告知用户，由用户决定是否重新运行 d-ask 修改需求，不得自行绕过或替换。

**正常执行（不打断）：**
- 创建新文件、新目录
- 编写代码、安装依赖
- 运行测试
- 小范围修改现有文件

**必须暂停，等待用户确认：**
- 删除或覆盖已有重要文件（非新建文件）
- 遇到真正的技术分叉，两种方案都合理，无法自行决策
- Task 验收失败，尝试修复 2 次后仍失败
- 上下文窗口接近上限

**暂停时的输出格式：**
```
⏸ 需要确认后继续

当前进度：Task X.X 已完成，下一个是 Task X.X
暂停原因：[具体原因]
需要你决定：[具体问题或选项]

确认后我将继续执行。
```

### 上下文窗口管理

每完成一个 Task 后，评估剩余上下文空间。

**接近上限时（约还剩 20% 空间）：**
```
⚠️ 上下文窗口即将满载

已完成：Task 1.1 ~ Task 2.3（已在 STEP.md 标记）
下一个：Task 2.4

请开启新对话，告诉 AI：
「读取 ASK.md 和 STEP.md，从下一个未完成的 Task 继续执行。」
```

### 完成后的输出报告

所有 Task 完成后输出：

```
✅ 所有 Task 执行完成

## 执行摘要
- 完成 Task 数：N
- 跳过 Task 数：N（如有，列出原因）

## 变更清单
[列出新增/修改的文件]

## 验收结果
[每个模块的验收状态]

## 建议下一步
可进行代码 review，重点关注：
- [需要人工审查的关键决策点]
- [有假设或不确定的地方]
```
