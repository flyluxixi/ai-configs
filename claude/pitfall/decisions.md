# 架构与决策类踩坑记录

## 2026-05-26 - 方案捆绑：A 让 C 变得不必要却没被识破，差点白做 17 处改动

**现象**: 提候选方案时把"A+C 必须一起做"作为推荐组合，但其实 A 已经独立解决了 C 想解决的问题，C 完全冗余。用户一句"代码需要改动吗"才点醒。如果用户没问，会照单全收做掉 C 的所有改动。
**根因**: 方案权衡时容易把"解决方案"和"想解决的问题"分开维度评估——A 解决 P1，C 解决 P2，看起来是两件独立的事，所以推 A+C。但实际上 P2 是 P1 的派生问题：A 把 P1 干掉后，P2 自动消失，C 失去存在理由。这种"派生依赖"在多方案权衡时容易看漏，尤其是 P1 和 P2 用不同视角描述时（如 P1="字段语义错"、P2="字段名撞词"，实际上字段一旦不存在，撞词问题自动消失）。
**解决**:
1. **核心检验**：提"A+C 组合"方案前，对每个组件问"如果只做 A，C 想解决的问题还存在吗？"。如果不存在，C 是冗余，从组合里删掉
2. **类比帮助识别**："吃饭前已经吃饱了还要不要吃零食"——独立的零食和饭，但饱了之后零食的必要性消失
3. **典型陷阱形态**：A 是"删除/移除某事物"、C 是"重命名/调整因该事物产生的副作用"——A 做完事物没了，副作用也没了，C 多余
4. **修正成本**：发现 C 多余时，如果决策已固化，要走"修正决策"（追加 d-decision 标注作废），而不是删除原决策——保留思考演化痕迹
