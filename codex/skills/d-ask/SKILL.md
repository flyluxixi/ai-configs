---
name: d-ask
description: 需求深度追问 skill。在动手编码之前，通过结构化对话把需求追问到零模糊点，确保 AI 完全理解每个模块的输入输出和验收标准，最终生成 ASK.md 供后续拆解 skill（d-step）使用。适用于 Claude Code、Codex 等 AI 编程工具。触发条件：用户说"追问我"、"问清楚再做"、"先别动手"、"你先问我"、"/d-ask"，或用户提出复杂开发任务但技术栈、模块边界、验收标准未明确。纯聊天、知识问答、单步简单任务不触发。
---

在动手之前，把「你以为你懂了」变成「你真的懂了」。

目标：基于已有项目文档，追问到每个模块的输入、输出、边界都无歧义，用户明确确认后写入 ASK.md，供 d-step 拆解使用。

---

## Step 1：预读项目文档

**优先从文档中获取信息，而不是问用户。**

按顺序读取以下文件（存在则读，不存在跳过）：
- `CLAUDE.md`
- `AGENTS.md`
- `README.md`
- `PROJECT_BACKGROUND.md`
- `ASK.md`（历史需求，检查是否需要更新）
- `ASK_DRAFT.md`（进行中的追问草稿）
- 其他项目根目录下的 `.md` 文档

读完后，在内部完成以下映射：

| 追问维度 | 文档中是否已覆盖 | 仍然模糊的点 |
|---------|--------------|------------|
| 目标 | | |
| 技术约束 | | |
| 模块边界 | | |
| 验收标准 | | |
| 异常处理 | | |
| 依赖 | | |

**只对文档中未覆盖或存在矛盾的点提问。**

然后告知用户：
```
📖 已读取项目文档（列出读到的文件名）。

基于文档，我已了解：
- [从文档中提取的关键信息摘要]

以下内容文档中未覆盖，需要确认：
□ [仍然模糊的问题1]
□ [仍然模糊的问题2]
...
```

如果文档已经覆盖所有维度，直接跳到 Step 5 复述确认，不再追问。

---

## Step 2：检查现有文件状态

**① 如果 `ASK_DRAFT.md` 存在（断点续追）：**

读取草稿，告知用户：
```
📋 发现未完成的追问草稿，已确认内容：
[已确认的维度摘要]

还有以下问题未确认：
□ [未确认项1]
□ [未确认项2]

从这里继续，还是从头开始？
```
- 用户选择继续 → 从草稿断点继续，进入 Step 4
- 用户选择从头开始 → 删除 ASK_DRAFT.md，继续 Step 3

**② 如果 `ASK.md` 存在（增量追问）：**

将已有需求内容作为基础，后续步骤只追问 Step 1 中识别出的缺口，完成后更新文件。→ 继续 Step 3

**③ 两个文件都不存在** → 继续 Step 3。

---

## Step 3：第一批——只问阻塞性问题

**判断标准：不弄清楚就无法开始开发的问题。**

对每个模糊点，先做判断：

> 「如果我现在对这个点做一个假设，能继续推进开发吗？」
> - 能 → 不问，自行假设，记录到 ASK_DRAFT.md 的「假设」区
> - 不能 → 列入第一批问题

**例外：以下维度无论是否能从文档或惯例推断，都必须纳入提问，不得自行假设：**
- 接口契约：入参/出参字段、格式、幂等性要求
- 认证与鉴权：token 来源、权限模型、失效处理
- 数据模型：核心字段与类型、状态机与状态转换
- 安全边界：哪些接口公开、哪些需要认证
- 外部依赖：具体 API 调用协议、错误处理约定

将第一批问题一次性列出，让用户统一回答：

```
以下问题不弄清楚无法开始开发，请逐一回答：

1. [阻塞性问题1]
2. [阻塞性问题2]
```

扫描维度（重点关注目标层和约束层，这里最容易出现阻塞性问题）：

1. **目标层** — 最终交付什么？成功的样子是什么？
2. **约束层** — 语言/框架、平台/环境、明确不能做的事
3. **模块层** — 每个模块的输入来源、输出去向、数据格式
4. **边界层** — 做到哪里算完？哪些明确不在范围内？
5. **依赖层** — 外部系统、API、服务及调用方式
6. **异常层** — 出错怎么处理？关键极端情况？
7. **验收层** — 怎么证明做对了？

**好问法 vs 差问法：**
- ✅ `这个接口是 REST 还是 GraphQL？`
- ❌ `你对技术有什么要求吗？`
- ✅ `用户鉴权：token 从前端传入还是从 session 读取？`
- ❌ `鉴权怎么做？`

## Step 4：后续批次——影响模块设计的问题 + 处理假设

用户回答第一批后，基于回答内容，识别出影响模块设计的新模糊点，再次一次性列出提问。

**对于能从上下文、文档、行业惯例推断的低风险问题，不提问，直接假设。**

低风险假设仅限于**格式约定类**，示例：
- 用户没说错误码格式 → 假设使用 HTTP 标准状态码
- 用户没说日志级别 → 假设 info/error 两级
- 用户没说时间戳格式 → 假设 ISO 8601

**以下运行语义类问题即使能从惯例推断，也不属于低风险，必须纳入高风险假设区，在 Step 5 显式确认：**
- 缓存：TTL、失效策略、一致性保证
- 重试：次数、间隔、退避策略
- 超时：连接超时、读取超时
- 幂等性：是否保证、如何实现
- 权限生效时间：变更后多久生效

**每轮追问结束后，将进度写入 `ASK_DRAFT.md`：**
```markdown
# 需求追问草稿（进行中）
> 最后更新：[时间]

## 已确认
[已清晰的维度和内容]

## 待确认
- [ ] [仍然模糊的问题]

## AI 自行假设（低风险，如有偏差请告知）
- [如：HTTP 标准状态码]
- [如：日志级别 info/error 两级]

## ⚠️ 临时假设（涉及接口/数据/安全，Step 5 需重点确认）
- [如：分页使用 cursor，影响 API 契约]
- [如：鉴权 token 从 Authorization Header 传入]
```

**遇到矛盾时即时点出**（不等到 Step 4.5）：
> 「你说"尽量快"，但又说"不引入缓存"——这两个有冲突，实际性能目标是什么？」
>
> Step 4.5 是在所有问题收集完毕后的全面检测；Step 4 中发现的矛盾就地解决，不积累到后面。

**终止条件（两个同时满足）：**
1. 所有阻塞性问题已清晰，剩余模糊点已有合理假设
2. 用户明确确认：「没问题了」「确认」「可以开始了」等

## Step 4.5：冲突与风险检测

在进入复述确认之前，对已收集的所有需求做一次主动检测。

**检测类型：**

**① 约束冲突**：两个要求在技术上无法同时满足
> 例：「低延迟实时响应」+「不允许缓存」+「不允许异步」

**② 规模冲突**：目标与资源不匹配
> 例：「单机部署」+「支持 10 万并发」

**③ 边界冲突**：做与不做的范围有交叉或矛盾
> 例：「不做用户管理」但「需要权限控制」

**④ 依赖风险**：依赖的外部系统存在明显隐患
> 例：依赖尚不稳定的第三方 API、依赖尚未存在的模块

**检测到冲突时，直接点出，不绕过：**

```
⚠️ 发现以下冲突，需要在继续前确认：

冲突1：[描述冲突]
  - 你要求：A
  - 你要求：B
  - 问题：A 和 B 同时满足意味着 [具体后果]
  - 请确认：[具体选择或优先级]
```

**没有冲突时**，静默跳过，直接进入 Step 5，不输出任何提示。

---

## Step 5：复述确认

所有维度清晰后，输出完整需求复述：

```
## 我的理解

**目标：**
[一句话，交付什么、给谁用]

**技术约束：**
- 语言 / 框架：
- 平台 / 环境：
- 禁止事项：

**模块与数据流：**
| 模块 | 输入（来源 + 格式） | 输出（去向 + 格式） |
|------|------------------|------------------|
| 模块A | ... | ... |

**边界：**
- ✅ 做：
- ❌ 不做：

**异常处理：**
[关键异常场景及处理方式]

**依赖：**
[外部系统 / API / 服务]

**验收标准：**
- [ ] [可验证的条件1]
- [ ] [可验证的条件2]

**⚠️ 以下假设涉及接口、数据或安全边界，请重点确认（如有偏差请立即指出，否则将写入 ASK.md）：**
- [高风险假设1：具体内容]
- [高风险假设2：具体内容]
（无高风险假设时省略此区块）

**其他低风险假设（如有偏差请告知）：**
- [如：使用 HTTP 标准状态码]
```

结尾说：**「以上是我的完整理解，请确认是否准确？有任何偏差请直接指出。」**

- 平台 / 环境：
- 禁止事项：

## 模块与数据流
| 模块 | 输入（来源 + 格式） | 输出（去向 + 格式） |
|------|------------------|------------------|
| 模块A | ... | ... |

## 边界
- ✅ 做：
- ❌ 不做：

## 异常处理
[关键异常场景及处理方式]

## 依赖

## 验收标准

## AI 自行假设（用户已确认，如后续发现偏差请告知）

写入后删除 `ASK_DRAFT.md`。

告知用户：
`✅ 需求已确认，已写入 ASK.md。可运行 d-step 进行模块拆解。`

## 需求冻结机制

ASK.md 写入后即视为**需求冻结**。

**冻结后的规则：**

- 当前需求不再变动，d-step 和后续执行以此版本为准
- 新增需求、范围变更、功能追加，一律进入 `ASK_CHANGELOG.md`，不直接修改 ASK.md

**ASK_CHANGELOG.md 格式：**

```markdown
# 需求变更记录

## [日期] 变更 #1
**变更类型：** 新增 / 修改 / 删除
**内容：** [描述变更]
**影响范围：** [受影响的模块或 Task]
**状态：** 待评估 / 已合并 / 已拒绝
```

**当用户说「顺便再加个…」时：**

```
📝 已记录到 ASK_CHANGELOG.md。

当前需求已冻结，此变更不会影响正在执行的任务。
如需将此变更合并进当前需求，请在本轮开发完成后重新运行 d-ask。
```

**写完后停止。不要自动开始执行任何代码或拆解任务。**
