---
name: d-ask
description: 需求深度追问 skill。在动手编码之前，通过结构化对话把需求追问到零模糊点，确保 AI 完全理解每个模块的输入输出和验收标准，最终生成 ASK.md 供后续拆解 skill（d-step）使用。适用于 Claude Code、Codex 等 AI 编程工具。触发条件：用户说"追问我"、"问清楚再做"、"先别动手"、"你先问我"、"/d-ask"，或用户提出复杂开发任务但技术栈、模块边界、验收标准未明确。纯聊天、知识问答、单步简单任务不触发。
---

在动手之前，把「你以为你懂了」变成「你真的懂了」。

目标：追问到每个模块的输入、输出、边界都无歧义，用户明确确认后写入 ASK.md，供 d-step 拆解使用。适用于 Claude Code、Codex 等 AI 编程环境。

---

## Step 0：检查现有文件状态

按以下优先级依次检查：

**① 检查 `ASK.md` 是否存在：**
- **存在且内容与当前需求一致**（目标、模块边界、验收标准均已覆盖）→ 输出：`✅ ASK.md 已存在且需求清晰，无需重复追问。如需更新请告知。` 然后停止。
- **存在但有新模糊点或需求有变化** → 告知用户「发现以下新问题需要补充」，只追问增量部分，完成后更新文件。

**② 检查 `ASK_DRAFT.md` 是否存在（断点续追）：**
- **存在** → 读取草稿内容，告知用户：
  ```
  📋 发现未完成的追问草稿，已确认的内容如下：
  [列出已确认的维度和内容摘要]

  还有以下问题未确认：
  □ [未确认项1]
  □ [未确认项2]

  从这里继续追问，还是从头开始？
  ```
  根据用户选择：继续则从断点追问；重新开始则删除草稿后进入 Step 1。

**③ 两个文件都不存在** → 继续 Step 1。

---

## Step 1：扫描并公示模糊点

读完用户需求后，**先列出你要追问的内容**，让用户知道你在追什么：

```
我需要先确认以下几点：
□ [模糊点1]
□ [模糊点2]
□ [模糊点3]
```

扫描维度（从高到低优先级）：

1. **目标层** — 最终要交付什么？成功的样子是什么？
2. **模块层** — 系统由哪些模块组成？每个模块的输入从哪来、输出给谁、格式是什么？
3. **约束层** — 语言/框架、平台/环境、性能要求、明确不能做的事
4. **边界层** — 做到哪里算完？哪些明确不在范围内？
5. **异常层** — 出错怎么处理？哪些极端情况需要考虑？
6. **依赖层** — 依赖哪些外部系统、API、服务？调用方式是什么？
7. **验收层** — 怎么证明做对了？

---

## Step 2：持续追问，直到零模糊

每轮优先追问**影响方向**的问题，再追问**影响细节**的问题。不设轮数上限，不设每轮问题数上限——有几个模糊点就问几个，问完即问，直到全部清晰为止。

根据回答实时更新理解，只追问仍然模糊的部分，已经清晰的不重复问。

**好问法 vs 差问法：**
- ✅ `这个接口是 REST 还是 GraphQL？`
- ❌ `你对技术有什么要求吗？`
- ✅ `用户鉴权模块：token 从前端传入还是从 session 读取？`
- ❌ `鉴权怎么做？`
- ✅ `失败时是静默忽略、返回错误码，还是抛出异常？`
- ❌ `错误处理怎么做？`

**遇到模糊或矛盾的回答：** 不要绕过，直接点出问题所在，继续追问直到清晰。例如：

> 「你说"尽量快"，但又说"不引入缓存"——这两个有冲突，实际上性能目标是什么？有没有具体的响应时间要求？」

**每轮追问结束后，将当前进度写入 `ASK_DRAFT.md`：**
```markdown
# 需求追问草稿（进行中）
> 最后更新：[时间]

## 已确认
[已清晰的维度和内容]

## 待确认
- [ ] [仍然模糊的问题1]
- [ ] [仍然模糊的问题2]
```
这样用户中途离开后，下次启动 d-ask 可从断点继续，不丢失进度。

**终止条件（两个都要同时满足）：**
1. AI 自身判断：每个模块的输入、输出、边界均无歧义
2. 用户明确确认：「没问题了」「确认」「可以开始了」等

---

## Step 3：复述确认

所有维度清晰后，输出完整的需求复述，让用户做最终确认：

```
## 我的理解

**目标：**
[一句话，交付什么、给谁用]

**技术约束：**
- 语言 / 框架：
- 平台 / 环境：
- 禁止事项：

**模块与数据流：**
| 模块 | 输入（来源 + 格式） | 输出（去向 + 格式） |
|------|------------------|------------------|
| 模块A | ... | ... |
| 模块B | ... | ... |

**边界：**
- ✅ 做：
- ❌ 不做：

**异常处理：**
[列出关键异常场景及处理方式]

**依赖：**
[外部系统 / API / 服务，及调用方式]

**验收标准：**
- [ ] [可验证的条件1]
- [ ] [可验证的条件2]
```

结尾说：**「以上是我的完整理解，请确认是否准确？有任何偏差请直接指出，我会继续追问。」**

---

## Step 4：用户确认后写入 ASK.md

用户明确确认无误后，将以下内容写入项目根目录的 `ASK.md`：

```markdown
# 需求确认文档

> 由 d-ask 生成，供 d-step 拆解使用。
> 最后更新：[日期时间]

## 目标
[一句话]

## 技术约束
- 语言 / 框架：
- 平台 / 环境：
- 禁止事项：

## 模块与数据流
| 模块 | 输入（来源 + 格式） | 输出（去向 + 格式） |
|------|------------------|------------------|
| 模块A | ... | ... |

## 边界
- ✅ 做：
- ❌ 不做：

## 异常处理
[关键异常场景及处理方式]

## 依赖
[外部系统 / API / 服务]

## 验收标准
- [ ] [可验证的条件]
```

写入 ASK.md 后，删除 `ASK_DRAFT.md`（草稿使命完成）。

告知用户：
`✅ 需求已确认，已写入 ASK.md。可运行 d-step 进行模块拆解。`

> 若当前环境无法写入文件，将上述内容以代码块输出，提示用户手动保存为 ASK.md。

**写完后停止。不要自动开始执行任何代码或拆解任务。**
