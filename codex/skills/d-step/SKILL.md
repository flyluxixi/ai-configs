---
name: d-step
description: 项目原子化拆解 skill。将复杂项目拆解为最小可独立执行的原子任务，生成 STEP.md 供逐步执行。适用于 Claude Code、Codex 等 AI 编程工具。触发条件：用户说"拆解"、"分步做"、"帮我拆任务"、"/d-step"，或项目涉及多个模块/功能需要分步推进时。单文件修改、简单 bug 修复、单步任务不触发。
---

把「把整个项目扔给 AI」变成「每次只给 AI 一个刚好够的任务」。

## 执行步骤

### Step 0：检查现有文件状态

按以下优先级依次检查：

**① 检查 `STEP.md` 是否存在：**
- **存在且拆解仍然有效**（任务清单完整、无未覆盖的新需求）→ 输出：`✅ STEP.md 已存在且拆解有效，无需重复拆解。` 并列出当前未完成的 Task，然后停止。
- **存在但 `ASK.md` 有更新或用户提出新需求** → 告知用户「检测到需求变化，将更新以下模块」，只重新拆解受影响的部分，完成后覆盖更新 `STEP.md`。

**② 检查 `STEP_DRAFT.md` 是否存在（断点续拆）：**
- **存在** → 读取草稿内容，告知用户：
  ```
  📋 发现未完成的拆解草稿，已完成的模块如下：
  [列出已拆解的模块摘要]

  还有以下模块未拆解：
  □ [未拆解模块1]
  □ [未拆解模块2]

  从这里继续，还是从头开始？
  ```
  根据用户选择：继续则从断点推进；重新开始则删除草稿后进入 Step 1。

**③ 两个文件都不存在** → 继续 Step 1。

---

### Step 1：获取需求

优先读取项目根目录的 `ASK.md`。

**如果 `ASK.md` 不存在：** 主动告知用户：
```
⚠️ 未找到 ASK.md。建议先运行 d-ask 完成需求确认，拆解结果会更准确。
是否继续仅凭当前信息拆解？
```
若用户选择继续，则补问以下最小必要信息：
- 最终交付物是什么？
- 技术栈 / 语言 / 框架？
- 从零开始还是已有代码？
- 有哪些主要功能模块？

### Step 2：识别模块边界

按「高内聚低耦合」划分功能域：

1. 列出所有功能点（格式：动词 + 名词，如「创建用户」「发送通知」）
2. 按业务域或数据归属聚合成模块
3. 标注模块间依赖关系（箭头方向 = 执行顺序约束）：

```
模块 A ──► 模块 B ──► 模块 D
           模块 C ──► 模块 D
```

### Step 3：原子化拆解

将每个模块拆解为原子任务，每个任务必须满足：
- 可在单次 AI 对话中完成
- 输入明确（文件路径、接口、数据格式）
- 输出可验证（不是「写完了」，而是「通过了 XX 测试」）
- 与其他任务的接口契约已定义

粒度标准：每个 Task 必须同时满足以下三点：
- **单一职责**：只做一件事，职责边界清晰
- **单次对话可完成**：AI 在一个上下文窗口内能独立完成，不依赖未完成的其他 Task
- **输出可独立验证**：有明确的完成标志，不是「写完了」，而是「通过了 XX 测试 / 接口返回预期结果」

粒度参考：
- ❌ 太粗：「实现用户系统」（横跨多个职责，无法单次完成）
- ❌ 太细：「写一个 if 语句」（无独立验收意义）
- ✅ 合适：「实现 JWT 登录接口，返回 token，错误时返回 401」

### Step 4：持续写草稿，完成后写入 STEP.md

拆解过程中，**每完成一个模块的拆解后**，将当前进度写入 `STEP_DRAFT.md`：
```markdown
# 拆解草稿（进行中）
> 最后更新：[时间]

## 已拆解
[已完成拆解的模块及任务清单]

## 待拆解
- [ ] [尚未拆解的模块1]
- [ ] [尚未拆解的模块2]
```
这样用户中途中断后，下次启动 d-step 可从断点继续，不丢失进度。

全部模块拆解完成后，将完整结果写入项目根目录的 `STEP.md`，格式如下：

```markdown
# 项目拆解文档

## 最后更新
[日期时间]

## 模块总览
| 模块 | 描述 | 依赖 | 顺序 |
|------|------|------|------|
| auth | 用户认证 | — | 1 |
| user | 用户管理 | auth | 2 |

## 依赖关系
auth ──► user ──► notify

## 模块详情

### [模块名]
**职责：**[一句话]

**任务清单：**
- [ ] Task 1.1：[名称] — 输入：X，输出：Y，验收：Z
- [ ] Task 1.2：[名称] — 输入：X，输出：Y，验收：Z

**接口契约：**[暴露给其他模块的数据结构或函数签名]

---

## 执行顺序
- 推荐顺序：Task 1.1 → Task 1.2 → Task 2.1 → ...
- 可并行：[列出无依赖关系、可同时执行的任务]

## 执行说明
每个 Task 可在当前对话中直接说「执行 Task 1.1」，
也可开新对话并告知 AI 先读取 ASK.md 和 STEP.md 再开始。
每完成一个 Task，在 STEP.md 中将对应条目标记为 `[x]`。
```

写入 STEP.md 后，删除 `STEP_DRAFT.md`（草稿使命完成）。

写完后告诉用户：`✅ 拆解完成，已写入 STEP.md。可直接说「执行 Task 1.1」开始，或开新对话后让 AI 读取 ASK.md 和 STEP.md。`
