---
name: d-step
description: 项目原子化拆解与自动执行 skill。将复杂项目拆解为最小可独立执行的原子任务，生成 STEP.md 后等用户确认再连续执行，只在关键节点暂停。触发条件：用户明确说「/d-step」、「开始执行」、「拆解并执行」、「帮我自动完成」。不得根据上下文自行推断并触发。单文件修改、简单 bug 修复、单步任务不触发。
---

把「把整个项目扔给 AI」变成「每次只给 AI 一个刚好够的任务，然后自动跑完」。

---

## 核心约束（优先级最高）

**1. 代码写本机，除非用户明确要求否则不直接 SSH 写文件**
所有源代码、配置文件必须用 apply_patch 写在本机项目目录，通过 `git push` → 服务器 `git pull + 编译` 方式部署。除非用户明确要求，否则禁止通过 SSH heredoc 或重定向在服务器上创建或修改源文件。`git push` 及服务器部署操作须用户明确授权后才可执行，不随任务执行自动触发。

原因：SSH heredoc 写的文件只存在于服务器，不进本机、不进 git，版本失控；出问题后无法追溯、无法回滚。

**2. 命名与现有项目对齐**
创建新目录、服务名、数据库名等之前，必须先观察现有项目的命名风格（如 `ls /apps/`），严格对齐，不自行发明格式。

**3. STEP.md 写完后必须等待用户确认再执行**
Step 4 完成后展示完整计划，明确询问用户「确认后开始执行，还是需要调整？」，收到确认再进入 Step 5。

---

## Step 1：检查现有文件状态

**① 检查 `STEP.md` 是否存在：**
- **存在且有效** → 读取文件，找到第一个未完成的 Task，告知用户：
  ```
  📋 发现已有 STEP.md，当前进度：
  已完成：Task X.X ~ X.X
  下一个：Task X.X [任务名]

  直接继续执行，还是重新拆解？
  ```
  - 用户选择继续执行 → 跳到 **Step 5：自动执行**
  - 用户选择重新拆解 → 删除 STEP.md，继续 Step 2

**② 检查 `STEP_DRAFT.md` 是否存在（断点续拆）：**
- **存在** → 读取草稿，告知用户：
  ```
  📋 发现未完成的拆解草稿，已拆解模块：[已完成的模块列表]

  从断点继续，还是丢弃草稿重新拆解？
  ```
  - 用户选择继续 → 从断点继续拆解，完成后进入 Step 4 确认环节
  - 用户选择丢弃重拆 → 删除 STEP_DRAFT.md，继续 Step 2

**③ 两个文件都不存在** → 继续 Step 2。

---

## Step 2：获取需求

优先读取 `ASK.md`。同时读取 `CLAUDE.md`、`README.md`、`PROJECT_BACKGROUND.md` 补充上下文。

**如果存在 `ASK_CHANGELOG.md`**，读取并检查是否有「待评估」条目：

- **有「待评估」条目** → 停止，告知用户：
  ```
  ⚠️ ASK_CHANGELOG.md 中有待评估的需求变更，必须先处理后再拆解：
  [列出待评估条目]

  请通过 d-ask 将变更合并、拒绝或标记为延期，更新 ASK.md 后再运行 d-step。
  ```
- **无「待评估」条目** → 继续拆解。

**如果 `ASK.md` 不存在：**
```
⚠️ 未找到 ASK.md。建议先运行 d-ask 完成需求确认，拆解结果会更准确。
是否继续仅凭当前信息拆解？
```
若用户选择继续，必须补问以下信息再开始拆解：
- 交付物是什么（新项目 / 新功能 / 重构）
- 技术栈（语言、框架、数据库）
- 本机项目目录路径
- 服务器部署路径（观察现有项目命名对齐）
- 主要功能模块

---

## Step 3：识别模块边界

按「高内聚低耦合」划分功能域：

1. 列出所有功能点（格式：动词 + 名词，如「创建用户」「发送通知」）
2. 按业务域聚合成模块
3. 标注模块间依赖关系：

```
模块 A ──► 模块 B ──► 模块 D
           模块 C ──► 模块 D
```

---

## Step 4：原子化拆解，写入 STEP.md，等待确认

每个 Task 必须同时满足：
- **单一职责**：只做一件事
- **单次对话可完成**：AI 在一个上下文窗口内能独立完成
- **输出可独立验证**：有明确完成标志（「通过了 XX 测试」而不是「写完了」）

粒度参考：
- ❌ 太粗：「实现用户系统」
- ❌ 太细：「写一个 if 语句」
- ✅ 合适：「实现 JWT 登录接口，返回 token，错误时返回 401」

拆解过程中每完成一个模块，写入 `STEP_DRAFT.md` 保存进度。

全部模块拆解完成后，写入 `STEP.md`：

```markdown
# 项目拆解文档

> 由 d-step 生成。
> 最后更新：[日期时间]
> 当前执行进度：Task 0/N

## 模块总览
| 模块 | 描述 | 依赖 | 顺序 |
|------|------|------|------|
| auth | 用户认证 | — | 1 |
| user | 用户管理 | auth | 2 |

## 依赖关系
auth ──► user ──► notify

## 模块详情

### [模块名]
**职责：**[一句话]

**任务清单：**
- [ ] Task 1.1：[名称] — 输入：X，输出：Y，验收：Z
- [ ] Task 1.2：[名称] — 输入：X，输出：Y，验收：Z

**接口契约：**[暴露给其他模块的数据结构或函数签名]

---

## 执行顺序
推荐顺序：Task 1.1 → Task 1.2 → Task 2.1 → ...
可并行：[无依赖关系、可同时执行的任务]

## 暂停条件（遇到以下情况才停下来等待确认）
- 删除或覆盖已有重要文件
- 遇到真正的技术分叉（两种方案都合理，需要人决策）
- Task 验收失败且尝试修复 2 次后仍失败
- 上下文窗口接近上限（主动告知用户，保存进度，等待新对话）
```

**STEP.md 写完后，删除 STEP_DRAFT.md，然后停下来，向用户展示完整计划：**

```
📋 拆解完成，共 N 个 Task，分 M 个模块。

[简要列出各模块及 Task 数]

执行顺序：Task 1.1 → Task 1.2 → ...

确认后开始执行，还是需要调整？
```

**必须等用户明确回复「确认」「开始」「执行」或等效指令后，才进入 Step 5。**

---

## Step 5：自动执行

收到用户确认后开始执行。

### 执行循环

```
读取 STEP.md → 找第一个未完成 [ ] Task
→ 执行 Task（代码写本机，SSH 只执行命令）
→ 验收（运行测试 / 检查输出 / 验证接口）
→ 验收通过：将 [ ] 改为 [x]，更新 STEP.md 中的执行进度
→ 找下一个未完成 Task → 继续循环
→ 所有 Task 完成 → 输出完成报告
```

### 执行时的行为规范

**核心约束：禁止重新定义 ASK.md 中已确认的架构与边界。**
ASK.md 是需求基准，d-step 的职责是实现它，而不是优化或改进它。
如果执行过程中发现 ASK.md 的某个决策存在问题，必须暂停并告知用户，由用户决定是否重新运行 d-ask 修改需求，不得自行绕过或替换。

**正常执行（不打断）：**
- 在本机用 apply_patch 创建或修改文件
- 编写代码、安装依赖
- 运行测试
- SSH 查看日志等只读命令

**除非用户明确要求，否则禁止：**
- 通过 SSH heredoc / 重定向在服务器上写源代码或配置文件（原因：绕过 git，版本失控）
- 创建目录、服务名、数据库名前未确认命名风格
- 在获得 git push / deploy 明确授权前，SSH 执行编译、启动或重启服务（原因：push 未完成时服务器运行的是旧代码，编译/启动结果无意义）

**必须暂停，等待用户确认：**
- 删除或覆盖已有重要文件（非新建文件）
- 遇到真正的技术分叉，两种方案都合理，无法自行决策
- Task 验收失败，尝试修复 2 次后仍失败
- 上下文窗口接近上限

**暂停时的输出格式：**
```
⏸ 需要确认后继续

当前进度：Task X.X 已完成，下一个是 Task X.X
暂停原因：[具体原因]
需要你决定：[具体问题或选项]

确认后我将继续执行。
```

暂停时同步发送企业微信通知：

```bash
notify "⏸ [项目名] 需要确认：[暂停原因摘要]"
```

### 上下文窗口管理

每完成一个 Task 后，评估剩余上下文空间。

**接近上限时（约还剩 20% 空间）：**
```
⚠️ 上下文窗口即将满载

已完成：Task 1.1 ~ Task 2.3（已在 STEP.md 标记）
下一个：Task 2.4

请开启新对话，告诉 AI：
「读取 ASK.md 和 STEP.md，从下一个未完成的 Task 继续执行。」
```

同步发送企业微信通知：

```bash
notify "⚠️ [项目名] 上下文即将满载，请开启新对话继续 Task X.X"
```

### 完成后的输出报告

所有 Task 完成后输出：

```
✅ 所有 Task 执行完成

## 执行摘要
- 完成 Task 数：N
- 跳过 Task 数：N（如有，列出原因）

## 变更清单
[列出新增/修改的文件]

## 验收结果
[每个模块的验收状态]

## 建议下一步
可进行代码 review，重点关注：
- [需要人工审查的关键决策点]
- [有假设或不确定的地方]
```

输出报告后，执行以下命令发送企业微信通知（项目名从 CLAUDE.md 或 ASK.md 读取）：

```bash
notify "✅ [项目名] 所有 Task 执行完成，可以复测"
```
